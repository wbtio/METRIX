# توثيق نظام الإنجاز الكبير (Major Milestone)

هذا الملف يشرح بصورة شاملة شلون تشتغل عملية إضافة الإنجاز الكبير داخل METRIX، من لحظة اختيار المستخدم لهذا النمط داخل نافذة تسجيل التقدم، إلى التقييم، توليد الصورة، حفظ السجل، إضافة النقاط، وعرض الإنجاز لاحقاً داخل قسم التحديات/الإنجازات الكبيرة.

---

## 1. الفكرة العامة

نظام الإنجاز الكبير مخصص للحالات الاستثنائية فقط، مو للمهام اليومية الاعتيادية.

مثال:

- إذا المستخدم سوّى مهمة يومية عادية، المفروض يستخدم تسجيل التقدم العادي.
- إذا المستخدم سوّى قفزة كبيرة جداً مقارنة بمهامه المعتادة، يقدر يسجلها كـ “إنجاز كبير”.

الذكاء الاصطناعي يقارن الإنجاز المدخل مع المهام الاعتيادية للهدف، وإذا شاف الإنجاز يستحق، يقبله ويصنفه إلى مستوى معيّن ويحسب له نقاط كبيرة.

## 3. مسار المستخدم داخل الواجهة

### 3.1 فتح نافذة تسجيل التقدم

داخل `Dashboard.tsx` يوجد زر تسجيل التقدم. عند الضغط عليه يتم فتح:

`ProgressLogDialog`

داخل هذه النافذة يظهر للمستخدم أكثر من نمط تسجيل:

1. تسجيل بالذكاء الاصطناعي.
2. تسجيل يدوي.
3. إنجاز كبير (Milestone).

زر الإنجاز الكبير يظهر بتصميم بنفسجي وفيه أيقونة كأس `Trophy`، والنص العربي الحالي تقريباً:

- العنوان: `إنجاز كبير (Milestone)`
- الوصف: `سجل إنجازاً استثنائياً كبيراً لتحصل على مكافأة ضخمة`

---

## 4. شاشة إدخال الإنجاز الكبير

عند اختيار المستخدم “إنجاز كبير”، يتم تغيير حالة النافذة:

- `mode = 'milestone'`

وتظهر شاشة خاصة تحتوي على:

### 4.1 الهيدر

- عنوان الشاشة: `إنجاز كبير (Milestone)`
- اسم الهدف الحالي.
- زر إغلاق يرجع المستخدم إلى شاشة اختيار نمط التسجيل.

### 4.2 رسالة تحذيرية

توجد رسالة واضحة تخبر المستخدم أن الإنجاز يجب أن يكون قفزة ضخمة مقارنة بالمهام العادية، وإلا سيتم رفضه من الذكاء الاصطناعي.

الغرض منها منع استخدام هذا النمط كبديل لتسجيل المهام اليومية.

### 4.3 حقول الإدخال

| الحقل | الحالة البرمجية | الوصف |
|---|---|---|
| اسم الإنجاز | `milestoneName` | اسم مختصر مثل “إطلاق التطبيق” |
| وصف الإنجاز | `logText` | شرح تفصيلي ليش هذا الإنجاز يعتبر قفزة استثنائية |
| أبعاد الصورة | `milestoneAspectRatio` | يختار المستخدم `1:1` أو `16:9` أو `9:16` |

### 4.4 زر الإرسال

زر الإرسال لا يتفعل إلا إذا:

- `milestoneName` غير فارغ.
- `logText` غير فارغ.
- `loading` ليس فعالاً.

عند الضغط يتم تنفيذ الدالة:

`handleMilestoneSubmit`

## 5. البيانات التي ترسل من الواجهة إلى الـ API

عند الإرسال، الواجهة ترسل طلب `POST` إلى:

`/api/goal/milestone`

البيانات المرسلة:

| الحقل | المصدر | الوصف |
|---|---|---|
| `goalId` | `goal.id` | معرف الهدف |
| `userInput` | `logText` | وصف الإنجاز الكبير |
| `milestoneName` | `milestoneName` | اسم الإنجاز |
| `targetPoints` | `goal.target_points` | نقاط الهدف النهائية |
| `dailyCap` | `calculateDailyCap(tasks)` | الحد اليومي المحسوب من المهام |
| `aspectRatio` | `milestoneAspectRatio` | أبعاد الصورة التذكارية |
| `tasksDescriptions` | `tasks.map(...)` | أسماء المهام الموجودة في الهدف |
| `goalContext` | `goal` | معلومات الهدف |
| `language` | لغة الواجهة | عربي أو إنكليزي |

## 6. ما الذي يحدث داخل API عند الإضافة؟

الملف المسؤول:

`src/app/api/goal/milestone/route.ts`

### 6.1 التحقق من الحقول المطلوبة

إذا نقص واحد من هذه الحقول:

- `goalId`
- `userInput`
- `milestoneName`
- `targetPoints`
- `dailyCap`

يرجع API خطأ `400`.

ملاحظة مهمة: إذا `targetPoints` أو `dailyCap` قيمتها `0`، الشرط الحالي يعتبرها ناقصة لأن الفحص يستخدم `!targetPoints` و `!dailyCap`.

### 6.2 التحقق من تسجيل الدخول

يستخدم السيرفر:

`supabase.auth.getUser()`

إذا لا يوجد مستخدم، يرجع `401 Authentication required`.

### 6.3 التحقق من ملكية الهدف

يتم البحث في جدول `goals` بشرطين:

- `id = goalId`
- `user_id = user.id`

إذا الهدف غير موجود أو لا يعود للمستخدم، يرجع `404`.

### 6.4 حد الإنجازات الكبيرة لكل هدف

قبل إضافة إنجاز جديد، يتم جلب سجلات الهدف من جدول:

`daily_logs`

ثم يتم عدّ السجلات التي تحتوي على:

`breakdown.milestone`

إذا العدد وصل إلى `2`، يتم رفض الإضافة برسالة:

`لا يمكن تسجيل أكثر من إنجازين كبيرين لكل هدف...`

إذن الحد الحالي هو:

- إنجازان كبيران فقط لكل هدف.

## 7. تقييم الإنجاز بالذكاء الاصطناعي

بعد التحقق، يستدعي API:

`GeminiService.evaluateMilestone(...)`

هذه الدالة موجودة في:

`src/lib/gemini.ts`

### 7.1 مهمة الذكاء الاصطناعي

الذكاء الاصطناعي يقرر:

- هل الإنجاز مقبول أم مرفوض؟
- إذا مقبول، شنو درجته؟
- شنو الاسم الملحمي المقترح؟
- شنو الوصف المختصر؟
- شنو رسالة التشجيع أو سبب الرفض؟

### 7.2 قاعدة القبول

حسب الـ prompt الحالي:

- إذا الجهد يعادل `3x` أو أكثر من المهام العادية: يقبل.
- إذا أقل من `3x`: يرفض.

### 7.3 مستويات الإنجاز

| المستوى | المعنى |
|---|---|
| `minor` | إنجاز صغير/مقبول، تقريباً 3x إلى 4x |
| `major` | إنجاز كبير، تقريباً 5x إلى 9x |
| `legendary` | إنجاز أسطوري، 10x أو أكثر |

### 7.4 شكل نتيجة التقييم

الدالة ترجع JSON يحتوي على:

| الحقل | الوصف |
|---|---|
| `is_milestone_accepted` | هل الإنجاز مقبول؟ |
| `rejection_reason` | سبب الرفض إذا مرفوض |
| `milestone_tier` | المستوى: minor / major / legendary |
| `suggested_base_points_multiplier` | مضاعف نقاط مقترح |
| `generated_name` | اسم مقترح للإنجاز |
| `short_description` | وصف مختصر |
| `coaching_message` | رسالة تشجيعية أو تفسير الرفض |

---

## 8. ماذا يحدث إذا تم رفض الإنجاز؟

إذا رجع الذكاء الاصطناعي:

`is_milestone_accepted = false`

فالـ API يرجع استجابة بحالة `200` لكن بالمحتوى:

- `success: false`
- `status: 'rejected'`
- `reason`
- `message`
- `coaching_message`

الواجهة تعرض الرسالة كتنبيه خطأ داخل شاشة الإنجاز الكبير، ولا يتم حفظ أي سجل، ولا تتم إضافة أي نقاط.

---

## 9. حساب نقاط الإنجاز الكبير

إذا تم قبول الإنجاز، يتم حساب النقاط حسب المستوى.

### 9.1 المضاعفات والحد الأعلى

| المستوى | المضاعف الافتراضي | الحد الأعلى من نقاط الهدف |
|---|---:|---:|
| `minor` | `3 × dailyCap` | 5% من `targetPoints` |
| `major` | `5 × dailyCap` | 10% من `targetPoints` |
| `legendary` | `10 × dailyCap` | 20% من `targetPoints` |

### 9.2 معادلة الحساب

الحساب الحالي:

1. `calculatedScore = suggested_base_points_multiplier × dailyCap`
2. `maxScore = floor(maxPercentage × targetPoints)`
3. `finalScore = min(calculatedScore, maxScore)`

يعني حتى لو الذكاء الاصطناعي اقترح مضاعف عالي، النقاط لا تتجاوز نسبة معينة من نقاط الهدف.

---

## 10. توليد الصورة التذكارية

بعد حساب النقاط، يحاول النظام توليد صورة تذكارية.

الدالة المسؤولة:

`GeminiService.generateMilestoneImage(...)`

### 10.1 النموذج المستخدم

يستخدم:

`gemini-2.5-flash-image`

### 10.2 طبيعة الصورة

الـ prompt يطلب صورة:

- مجردة بالكامل.
- بأسلوب Cyberpunk / Matrix.
- بدون نصوص أو أرقام أو كلمات داخل الصورة.
- تمثل مستوى الإنجاز وسياقه.

### 10.3 رفع الصورة إلى Supabase Storage

إذا تولدت الصورة بنجاح:

1. يتم تحويل base64 إلى Buffer.
2. يتم إنشاء اسم ملف بالشكل:

`milestone-${goalId}-${Date.now()}.jpg`

3. يتم رفع الصورة إلى bucket اسمه:

`milestones`

4. يتم أخذ رابط عام للصورة:

`publicUrl`

ويحفظ لاحقاً داخل `breakdown.milestone.imageUrl`.

### 10.4 فشل الصورة لا يمنع الحفظ

إذا فشل توليد الصورة أو رفعها، النظام فقط يكتب warning في الـ console، ويكمل حفظ الإنجاز بدون صورة.

في هذه الحالة تكون:

`imageUrl = null`

## 11. شكل الحفظ داخل قاعدة البيانات

الإنجاز الكبير لا يحفظ في جدول منفصل حالياً.

هو يحفظ كسجل داخل جدول:

`daily_logs`

### 11.1 الحقول الأساسية في `daily_logs`

عند القبول، يتم إدخال سجل جديد يحتوي على:

| الحقل | القيمة |
|---|---|
| `goal_id` | معرف الهدف |
| `user_input` | وصف الإنجاز الكامل |
| `ai_score` | النقاط النهائية `finalScore` |
| `ai_feedback` | رسالة الذكاء الاصطناعي |
| `breakdown` | يحتوي معلومات الإنجاز الكبير |

### 11.2 شكل `breakdown`

يحفظ بهذا الشكل المنطقي:

- `version: 2`
- `items: []`
- `milestone`:
  - `tier`
  - `imageUrl`
  - `name`
  - `description`
  - `short_description`

المهم هنا أن وجود `breakdown.milestone` هو العلامة التي تميّز أن هذا السجل إنجاز كبير، وليس تسجيل تقدم عادي.

---

## 12. إضافة النقاط إلى الهدف

بعد حفظ سجل `daily_logs`، يتم استدعاء دالة قاعدة البيانات:

`increment_goal_points`

بالقيم:

- `goal_uuid = goalId`
- `points_to_add = finalScore`

هذه الخطوة ترفع `current_points` للهدف.

إذا فشلت هذه الخطوة بعد إدخال السجل، حالياً سيرجع API خطأ، لكن السجل قد يكون تم إدخاله قبل فشل زيادة النقاط. هذه نقطة مهمة للمراجعة مستقبلاً.

## 13. استجابة API عند النجاح

عند نجاح العملية، يرجع API:

| الحقل | الوصف |
|---|---|
| `status` | `ok` |
| `tier` | مستوى الإنجاز |
| `score` | النقاط المضافة |
| `imageUrl` | رابط الصورة أو null |
| `message` | رسالة الذكاء الاصطناعي |
| `logId` | معرف سجل `daily_logs` |

## 14. ماذا تفعل الواجهة بعد النجاح؟

بعد استلام نجاح الإضافة داخل `handleMilestoneSubmit`:

1. تنشئ نتيجة عرض مؤقتة داخل `evaluation`.
2. تضيف النقاط ذهنياً إلى نقاط الهدف الحالية لمعرفة هل الهدف اكتمل.
3. إذا اكتمل الهدف، تعرض احتفال الإنجاز.
4. إذا لم يكتمل، تنتقل إلى شاشة نتيجة مشابهة لشاشة نتيجة تقييم AI.
5. ترسل حدث عام للواجهة:

`challenge-log-updated`

هذا الحدث يخلي أجزاء ثانية مثل التحديات تعيد الجلب وتعرض الإنجاز الجديد.

## 15. عرض الإنجازات الكبيرة

الإنجازات الكبيرة تظهر داخل قسم التحدي/الإنجازات الكبيرة.

الملف الأساسي:

`src/components/challenge/ActivityCard.tsx`

### 15.1 من أين تأتي البيانات؟

واجهة التحدي تستدعي API:

`src/app/api/challenges/by-goal/route.ts`

داخل هذا API يتم جلب `daily_logs`، وكل سجل يتم فحصه:

إذا كان يحتوي على `breakdown.milestone`، يتم تمريره كـ milestone event.

### 15.2 فلترة العرض

داخل `ActivityCard` يتم تنفيذ:

`recentEvents.filter(e => e.milestone)`

يعني البطاقة تعرض فقط السجلات التي تحتوي على milestone.

### 15.3 تصميم بطاقة الإنجاز

كل بطاقة تعرض:

- صورة الإنجاز إذا موجودة.
- إذا لا توجد صورة، تعرض أيقونة حسب المستوى.
- عدد النقاط `+points`.
- مستوى الإنجاز بلون مختلف.
- اسم الإنجاز.
- وصف مختصر.
- قائمة خيارات إذا الإنجاز يعود للمستخدم الحالي.

### 15.4 ألوان المستويات

| المستوى | التصميم |
|---|---|
| `minor` | أزرق / بنفسجي خفيف |
| `major` | بنفسجي / فوشي |
| `legendary` | ذهبي / برتقالي |

## 16. تعديل الإنجاز بعد الحفظ

من بطاقة الإنجاز في `ActivityCard`، المستخدم يمكنه:

- تعديل الاسم.
- تعديل الوصف.
- تغيير اللون/المستوى الظاهري.
- حذف الإنجاز.

### 16.1 تعديل الاسم أو الوصف أو المستوى

يتم إرسال طلب `PATCH` إلى:

`/api/goal/milestone`

الـ API يتحقق من:

1. وجود `logId` و `goalId`.
2. المستخدم مسجل دخول.
3. الهدف يعود للمستخدم.
4. السجل موجود وينتمي لهذا الهدف.
5. السجل يحتوي على `breakdown.milestone`.

ثم يحدث فقط `breakdown.milestone`.

ملاحظة مهمة: تغيير `tier` من الواجهة يغير اللون/التصنيف المحفوظ، لكنه لا يعيد حساب النقاط.

---

## 17. حذف الإنجاز

عند حذف الإنجاز، يتم إرسال طلب `DELETE` إلى:

`/api/goal/milestone`

### 17.1 خطوات الحذف

1. التحقق من المستخدم.
2. التحقق من ملكية الهدف.
3. جلب سجل `daily_logs`.
4. قراءة `ai_score` حتى يتم خصم النقاط.
5. إذا توجد صورة، يحاول حذفها من bucket `milestones`.
6. حذف سجل `daily_logs`.
7. خصم النقاط من الهدف عبر:

`increment_goal_points` بقيمة سالبة.

### 17.2 نتيجة الحذف

يرجع API:

- `status: 'ok'`
- `revertedScore`

ثم ترسل الواجهة حدث تحديث حتى يتم تحديث العرض.

---

## 18. نقاط القوة الحالية

- النظام يفرق بين الإنجاز العادي والإنجاز الكبير.
- يوجد تحقق من ملكية الهدف قبل الإضافة أو التعديل أو الحذف.
- يوجد حد أقصى: إنجازان كبيران لكل هدف.
- الذكاء الاصطناعي يرفض الإنجازات الاعتيادية.
- النقاط محكومة بسقف حسب `targetPoints` حتى لا تكون مبالغ بها.
- الصورة اختيارية، وفشلها لا يكسر الحفظ.
- الحذف يرجع النقاط ويحاول حذف الصورة.
- العرض داخل `ActivityCard` جميل ومضغوط ومناسب للإنجازات الكبيرة.

---

## 19. ملاحظات ومخاطر للمراجعة المستقبلية

### 19.1 لا يوجد جدول منفصل للإنجازات الكبيرة

حالياً الإنجاز الكبير محفوظ داخل `daily_logs` عن طريق `breakdown.milestone`.

هذا عملي، لكن إذا النظام كبر، قد يكون جدول مستقل مثل `milestones` أو `goal_milestones` أوضح وأسهل للفهرسة والاستعلام.

### 19.2 إدخال السجل وزيادة النقاط ليست عملية واحدة Transaction

الـ API يدخل سجل `daily_logs` ثم يستدعي `increment_goal_points`.

إذا نجح الإدخال وفشلت زيادة النقاط، ممكن يصير عدم تطابق بين السجلات والنقاط.

الأفضل مستقبلاً جعل العملية داخل RPC واحدة أو Transaction.

### 19.3 تغيير المستوى لا يعيد حساب النقاط

من الواجهة يمكن تغيير لون/مستوى البطاقة عبر `PATCH`، لكن النقاط لا تتغير.

هذا جيد إذا كان التغيير تجميلي فقط، لكن إذا المستوى له معنى حسابي، لازم إما:

- منع تغيير المستوى بعد الحفظ.
- أو إعادة حساب النقاط عند تغييره.

### 19.4 شرط `targetPoints` و `dailyCap`

الـ API يعتبر `0` قيمة ناقصة بسبب استخدام `!targetPoints` و `!dailyCap`.

إذا كان الهدف أو الاختبار يسمح بقيم صفرية، الأفضل استخدام تحقق أوضح مثل:

- `targetPoints == null`
- `dailyCap == null`

### 19.5 حذف الصورة يعتمد على استخراج اسم الملف من الرابط

الحذف يحاول استخراج اسم الملف من `imageUrl`، وهذا يعمل غالباً، لكن إذا تغير شكل رابط Supabase مستقبلاً قد يفشل.

الأفضل حفظ `storagePath` بجانب `imageUrl` داخل `breakdown.milestone`.

### 19.6 حد الإنجازات يعتمد على قراءة كل سجلات الهدف

حساب عدد الإنجازات يتم بجلب سجلات `daily_logs` ثم فلترتها في السيرفر.

إذا كثرت السجلات، الأفضل عمل Query أدق أو تخزين marker مستقل.

## 20. الخلاصة

عملية الإنجاز الكبير حالياً تسير بهذا التسلسل:

1. المستخدم يفتح تسجيل التقدم.
2. يختار إنجاز كبير.
3. يكتب اسم ووصف الإنجاز ويختار أبعاد الصورة.
4. الواجهة ترسل الطلب إلى `/api/goal/milestone`.
5. السيرفر يتحقق من المستخدم والهدف وحد الإنجازات.
6. Gemini يقيّم هل الإنجاز يستحق أم لا.
7. إذا مرفوض، لا يتم الحفظ.
8. إذا مقبول، يتم تحديد المستوى والنقاط.
9. يحاول النظام توليد صورة ورفعها.
10. يحفظ الإنجاز داخل `daily_logs` كـ `breakdown.milestone`.
11. يزيد نقاط الهدف عبر `increment_goal_points`.
12. الواجهة تعرض النتيجة وتحدث قسم التحديات.
13. الإنجاز يظهر كبطاقة داخل `ActivityCard`.
14. يمكن لاحقاً تعديل الاسم/الوصف/المستوى أو حذف الإنجاز وخصم نقاطه.

هذا النظام جيد كبداية، ومصمم بصورة واضحة، لكنه يستفاد كثيراً من تحسينات مستقبلية حول الـ Transaction، فصل جدول الإنجازات، وحفظ مسار الصورة في Storage.
